MCP vs API — What’s the Difference?

A plain-English guide with real code you can run.

Ritvik Gaur January 2026 6 min read

The Short Answer

📞

API

Calling a restaurant directly. You need their phone number, know their menu, and speak their language.

🔌

MCP

A food delivery app. One app, every restaurant, same interface. You don't need each restaurant's number.

API = a way for one program to talk to another program.
MCP = a universal way for AI to discover and use any tool.

MCP doesn't replace APIs. MCP servers usually wrap APIs. It's a standard layer on top that lets AI find and use tools on its own.

Why Does MCP Exist?

Say you're building an AI assistant. You want it to check the weather, search GitHub, and query a database. With APIs, you write separate custom code for each one — different URLs, different auth, different formats.

3 apps × 3 tools = 9 custom integrations.
10 apps × 10 tools = 100 custom integrations.

MCP fixes this. One standard protocol, and every app just speaks MCP.
10 apps + 10 tools = 20 connections. Done.

Show Me the Code

Let's build a weather tool both ways so you can see the difference.

The Old Way — call the API yourself

agent_old_way.py Python 
import httpx

def get_forecast(lat: float, lon: float) -> str:
    """You write ALL of this for every single API."""
    with httpx.Client() as client:
        points = client.get(
            f"https://api.weather.gov/points/{lat},{lon}",
            headers={"User-Agent": "app/1.0"},
        ).json()
        forecast = client.get(
            points["properties"]["forecast"],
            headers={"User-Agent": "app/1.0"},
        ).json()

    return forecast["properties"]["periods"][0]["detailedForecast"]

print(get_forecast(37.77, -122.42))

This works, but it's glued to one API. Want to add GitHub search? Write it all again. Different URL, different auth, different format. Repeat forever.

The MCP Way — build a server once, use it anywhere

server.py Python 
import httpx
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather")

@mcp.tool()
async def get_forecast(latitude: float, longitude: float) -> str:
    """Get weather forecast for a location."""
    async with httpx.AsyncClient() as client:
        points = await client.get(
            f"https://api.weather.gov/points/{latitude},{longitude}",
            headers={"User-Agent": "app/1.0"},
        )
        forecast_url = points.json()["properties"]["forecast"]
        forecast = await client.get(forecast_url, headers={"User-Agent": "app/1.0"})
    return forecast.json()["properties"]["periods"][0]["detailedForecast"]

if __name__ == "__main__":
    mcp.run(transport="stdio")

That's it. The @mcp.tool() decorator turns any function into a tool that any AI can discover and call. No hardcoded schemas. No manual dispatch. The AI just asks: "what tools do you have?" and the server answers.

How does the AI use it?

client.py Python 
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async with stdio_client(StdioServerParameters(
    command="python", args=["server.py"]
)) as (read, write):
    async with ClientSession(read, write) as session:
        await session.initialize()

        tools = await session.list_tools()      # "What can you do?"
        result = await session.call_tool(       # "Do this."
            "get_forecast",
            {"latitude": 37.77, "longitude": -122.42},
        )

This same code works with ANY MCP server. Swap the weather server for GitHub, a database, a file system — the client code doesn't change. That's the whole point.

The bottom line

APIs let software talk to software.
MCP lets AI talk to everything — through one universal plug.

MCP doesn't replace APIs. It gives AI a standard way to use them.

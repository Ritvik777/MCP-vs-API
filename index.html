MCP vs API — What’s the Difference?

A plain-English guide with real, runnable code. No API keys needed.

Ritvik Gaur January 2026 10 min read

The Short Answer

📞

API

Calling a restaurant directly. You need their phone number, know their menu, and speak their language.

🔌

MCP

A food delivery app. One app, every restaurant, same interface. You don't need each restaurant's number.

API = a way for one program to talk to another program.
MCP = a universal way for AI to discover and use any tool.

MCP doesn't replace APIs. MCP servers usually wrap APIs. It's a standard layer on top that lets AI find and use tools on its own.

Why Does MCP Exist?

Say you're building an AI assistant. You want it to check the weather, search GitHub, and query a database. With APIs, you write separate custom code for each one — different URLs, different auth, different formats.

3 apps × 3 tools = 9 custom integrations.
10 apps × 10 tools = 100 custom integrations.

MCP fixes this. One standard protocol, every app just speaks MCP.
10 apps + 10 tools = 20 connections. Done.

No API keys needed. MCP is just a protocol — a way for programs to talk. The MCP server and client we build below don't need any LLM key, OpenAI key, or Anthropic key. They work on their own. An AI host like Claude Desktop uses its own key separately — MCP doesn't care about that.

The Old Way — Call the API Yourself

Let's build a weather tool. First, the traditional way — calling the National Weather Service API directly:

agent_old_way.py Python 
import httpx

NWS_BASE = "https://api.weather.gov"
HEADERS = {"User-Agent": "weather-app/1.0"}

def get_forecast(latitude: float, longitude: float) -> str:
    """Fetch forecast — custom code for this one specific API."""
    with httpx.Client() as client:
        # Step 1: Get the forecast grid endpoint for this location
        points = client.get(
            f"{NWS_BASE}/points/{latitude},{longitude}",
            headers=HEADERS, timeout=30.0,
        ).json()

        # Step 2: Get the actual forecast from the grid endpoint
        forecast_url = points["properties"]["forecast"]
        forecast = client.get(
            forecast_url, headers=HEADERS, timeout=30.0,
        ).json()

    # Step 3: Format the first 3 periods into readable text
    periods = forecast["properties"]["periods"][:3]
    return "\n".join(
        f"{p['name']}: {p['temperature']}°{p['temperatureUnit']}, "
        f"{p['detailedForecast']}"
        for p in periods
    )

# Run it
print(get_forecast(37.7749, -122.4194))

Let's walk through this like reading a recipe:

  • import httpx — this loads a library that lets Python visit websites and get data back. Think of it as giving Python a web browser.
  • NWS_BASE = "https://api.weather.gov" — this is the address of the weather service. Like saving a restaurant's phone number in your contacts.
  • HEADERS = {"User-Agent": "weather-app/1.0"} — the weather service wants to know who's calling. This is us saying "hi, I'm weather-app". No password or key needed.
  • client.get(...).json() — this visits a web address and reads the response. .json() turns the raw response into a Python dictionary we can work with.
  • We make two requests: first we ask "where is this location on your grid?", then we ask "what's the forecast for that grid spot?"
  • periods[:3] — the forecast comes back in chunks (Today, Tonight, Tomorrow...). We grab the first 3.
  • print(get_forecast(37.7749, -122.4194)) — run it! Those numbers are San Francisco's latitude and longitude.

This works. But notice: everything is hardwired. The URL, the headers, the way we parse the response — it's all specific to this one weather service. Want to add GitHub search? You'd write a completely different function with different URLs, different auth, different response format. Every new tool means more custom code glued into your app.

The MCP Way — Build a Server

Now let's wrap that same weather API as an MCP server. The server is a standalone program that any AI can use — Claude, Cursor, your own agent, whatever.

Install the MCP SDK

terminal Shell 
pip install "mcp[cli]" httpx

The server (full code)

server.py Python 
from typing import Any
import httpx
from mcp.server.fastmcp import FastMCP

# Create an MCP server called "weather"
mcp = FastMCP("weather")

NWS_API_BASE = "https://api.weather.gov"
USER_AGENT = "weather-mcp-server/1.0"


async def make_nws_request(url: str) -> dict[str, Any] | None:
    """Helper: make a request to the NWS API."""
    headers = {"User-Agent": USER_AGENT, "Accept": "application/geo+json"}
    async with httpx.AsyncClient() as client:
        try:
            response = await client.get(url, headers=headers, timeout=30.0)
            response.raise_for_status()
            return response.json()
        except (httpx.HTTPError, ValueError):
            return None


@mcp.tool()
async def get_forecast(latitude: float, longitude: float) -> str:
    """Get weather forecast for a location.

    Args:
        latitude: Latitude of the location
        longitude: Longitude of the location
    """
    points_url = f"{NWS_API_BASE}/points/{latitude},{longitude}"
    points_data = await make_nws_request(points_url)

    if not points_data:
        return "Unable to fetch forecast data for this location."

    forecast_url = points_data["properties"]["forecast"]
    forecast_data = await make_nws_request(forecast_url)

    if not forecast_data:
        return "Unable to fetch detailed forecast."

    periods = forecast_data["properties"]["periods"]
    return "\n---\n".join(
        f"{p['name']}:\n"
        f"  Temperature: {p['temperature']}°{p['temperatureUnit']}\n"
        f"  Wind: {p['windSpeed']} {p['windDirection']}\n"
        f"  {p['detailedForecast']}"
        for p in periods[:5]
    )


@mcp.tool()
async def get_alerts(state: str) -> str:
    """Get active weather alerts for a US state.

    Args:
        state: Two-letter US state code (e.g. CA, NY)
    """
    url = f"{NWS_API_BASE}/alerts/active/area/{state}"
    data = await make_nws_request(url)

    if not data or "features" not in data:
        return "Unable to fetch alerts or no alerts found."

    if not data["features"]:
        return "No active alerts for this state."

    alerts = []
    for feature in data["features"]:
        props = feature["properties"]
        alerts.append(
            f"Event: {props.get('event', 'Unknown')}\n"
            f"Area: {props.get('areaDesc', 'Unknown')}\n"
            f"Severity: {props.get('severity', 'Unknown')}"
        )
    return "\n---\n".join(alerts)


# Start the server
if __name__ == "__main__":
    mcp.run(transport="stdio")

Let's walk through what each part does:

  • from mcp.server.fastmcp import FastMCP — loads the MCP toolkit. This is what turns a normal Python script into a server that AI can talk to.
  • mcp = FastMCP("weather") — creates a server and gives it a name: "weather". This is like naming your restaurant so the delivery app can list it.
  • @mcp.tool() — this is the magic line. Put it above any function and that function becomes a tool any AI can find and use. You don't need to write any extra configuration — the SDK reads your function name, its inputs, and the description, and creates the "menu" automatically.
  • async def get_forecast(latitude: float, longitude: float) — the function itself. The latitude: float part tells the SDK "this input should be a number". The AI will see that and know what to pass in.
  • """Get weather forecast for a location.""" — this description is important! It's what the AI reads to decide "is this the right tool for the user's question?"
  • make_nws_request — just a helper that visits the weather website. It's the exact same API logic as the old way — nothing MCP-specific about it.
  • get_forecast and get_alerts — two tools on our "menu". Any MCP client can discover both and call either one.
  • mcp.run(transport="stdio") — opens for business! The server starts listening. "stdio" means it talks through standard input/output — that's how Claude Desktop and Cursor communicate with local servers.

Notice what's NOT here: no OpenAI import, no Anthropic import, no API key, no mention of any specific AI. This server has no idea who will use it. It just puts tools on the menu and waits. Claude can use it. Cursor can use it. Your own app can use it. That's the whole point.

The Client — How AI Finds and Uses Tools

Now let's write a client that connects to our server, discovers the tools, and calls them. This is what AI hosts like Claude Desktop do under the hood:

client.py Python 
import asyncio
import sys
from pathlib import Path
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

SERVER_SCRIPT = Path(__file__).parent / "server.py"

async def main():
    print("Connecting to weather MCP server...\n")

    # Tell the client how to start the server
    server_params = StdioServerParameters(
        command=sys.executable,          # use current Python
        args=[str(SERVER_SCRIPT)],       # run server.py
    )

    # Connect to the server
    async with stdio_client(server_params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # Step 1: Ask "what tools do you have?"
            tools = await session.list_tools()
            print("Available tools:")
            for tool in tools.tools:
                print(f"  - {tool.name}: {tool.description}")

            # Step 2: Call a tool by name
            print("\nFetching forecast for San Francisco...\n")
            result = await session.call_tool(
                "get_forecast",
                arguments={"latitude": 37.7749, "longitude": -122.4194},
            )
            print(result.content[0].text)

asyncio.run(main())

Think of this as the delivery app placing an order:

  • StdioServerParameters(command=..., args=[...]) — tells the client "the server is this Python file, here's how to start it." Like telling the delivery app where the restaurant is.
  • stdio_client(server_params) — actually starts the server and opens a connection. The restaurant is now open.
  • session.initialize() — the client and server say hello and agree on which version of MCP to speak. Like a handshake.
  • session.list_tools() — the client asks: "hey, what's on the menu?" The server replies with every tool it has, what each tool does, and what inputs it needs. The client didn't know any of this in advance.
  • session.call_tool("get_forecast", arguments={...}) — the client places an order: "run get_forecast with these coordinates." The server runs the function and sends back the result.
  • result.content[0].text — the response from the server. We just print it.

This is the whole point: this same client code works with any MCP server. Swap the weather server for a GitHub server, a database server, a file-system server — the client code doesn't change. The protocol is the same.

How Does Claude Know Which Tool to Use?

When Claude Desktop starts, it connects to your server and asks one question: "What tools do you have?" Your server replies with this:

server response (auto-generated) JSON 
{
  "name": "get_forecast",
  "description": "Get weather forecast for a location.",
  "inputSchema": {
    "properties": {
      "latitude": { "type": "number" },
      "longitude": { "type": "number" }
    }
  }
}

You never wrote that JSON. The SDK generated it from your Python code:

server.py Python 
@mcp.tool()
async def get_forecast(latitude: float, longitude: float) -> str:
    """Get weather forecast for a location."""
  • The function name (get_forecast) became the tool name.
  • The docstring ("Get weather forecast...") became the description.
  • The type hints (float) became the parameter schema.

Now when you ask Claude "What's the weather in New York?", two things happen:

  1. MCP handles discovery — it tells Claude a tool called get_forecast exists and what it accepts.
  2. Claude (the LLM) handles selection — it reads the tool name and description, decides it matches your question, figures out the latitude/longitude for New York, and calls the tool.

This is why good names and docstrings matter. They're literally what the AI reads to decide which tool to pick. A tool called xyz123 with no description? Claude won't know when to use it.

Use It in Claude Desktop

Want to try this yourself? Here's how to connect our weather server to Claude Desktop in 3 steps:

Step 1: Add the server to Claude's config

Open (or create) the file claude_desktop_config.json:

  • Mac: ~/Library/Application Support/Claude/claude_desktop_config.json
  • Windows: %APPDATA%\Claude\claude_desktop_config.json

Paste this in:

claude_desktop_config.json JSON 
{
  "mcpServers": {
    "weather": {
      "command": "uv",
      "args": [
        "run", "--with", "mcp[cli]", "--with", "httpx",
        "python", "/path/to/your/server.py"
      ]
    }
  }
}

Replace /path/to/your/server.py with the actual path to the file. The uv run --with part auto-installs the dependencies so Claude can find them.

Step 2: Restart Claude Desktop

Quit (Cmd+Q on Mac) and reopen. Claude will launch your server and discover the tools.

Step 3: Ask a weather question

Type something like: "What's the weather in San Francisco?"

Claude will see your get_forecast tool, call it with the right coordinates, and show you the live forecast — using your server, with zero extra code.

That's the power of MCP: You built the server once, tested it with client.py, and now Claude Desktop uses the exact same server. No integration code. No SDK changes. Just one config line.

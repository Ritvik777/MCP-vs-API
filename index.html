The Integration Problem

Imagine you're building an AI assistant. You want it to check the weather, search a database, look up a user's calendar, and read files on disk. With traditional APIs, every one of these integrations requires bespoke glue code — different auth mechanisms, different request formats, different error handling, different docs to read.

Now multiply that by every AI application in the world. Every developer is writing the same kind of integration code, over and over, for every tool their model needs to use. The result is a fragile web of point-to-point connections that's expensive to build and painful to maintain.

This is the M×N problem: M applications each integrating with N tools creates M×N unique integrations. What if there was a single, universal standard that turned M×N into M+N?

The core insight: We don't need more APIs. We need a standard protocol that lets AI models discover and use any tool through a single interface — the way USB-C replaced a drawer full of cables.

What Is MCP?

The Model Context Protocol (MCP), open-sourced by Anthropic in late 2024, is a standard that defines how AI applications communicate with external tools and data sources. Think of it as a universal plugin system for AI.

MCP uses a client-server architecture built on JSON-RPC 2.0. The moving parts are:

  • MCP Host — the AI application (Claude Desktop, Cursor, your custom agent)
  • MCP Client — lives inside the host, manages connections to servers
  • MCP Server — a lightweight process that exposes capabilities to the AI

Each server can expose three types of capabilities:

  • Tools — functions the AI can execute (query a database, call an API, run a script)
  • Resources — data the AI can read (files, logs, configuration)
  • Prompts — reusable interaction templates

The key insight: the AI discovers available tools at runtime. It doesn't need hardcoded knowledge of what's available — it asks the server "what can you do?" and gets back a structured list of tools with their schemas. This is fundamentally different from traditional API integration, where everything must be wired up in advance by a developer.

The Old Way — Direct API Integration

Let's make this concrete. Suppose we want our AI agent to fetch weather forecasts. Here's what a traditional integration looks like — calling the National Weather Service API directly inside an OpenAI function-calling agent:

agent_old_way.py Python 
import httpx
import json
from openai import OpenAI

NWS_BASE = "https://api.weather.gov"
client = OpenAI()

# Hardcoded tool definition — tightly coupled to this one API
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_forecast",
            "description": "Get weather forecast for a latitude/longitude",
            "parameters": {
                "type": "object",
                "properties": {
                    "latitude":  {"type": "number"},
                    "longitude": {"type": "number"},
                },
                "required": ["latitude", "longitude"],
            },
        },
    }
]

def get_forecast(latitude: float, longitude: float) -> str:
    """Fetch forecast from NWS — custom code for this specific API."""
    headers = {"User-Agent": "weather-app/1.0"}
    with httpx.Client() as http:
        points = http.get(
            f"{NWS_BASE}/points/{latitude},{longitude}",
            headers=headers, timeout=30.0,
        ).json()
        forecast_url = points["properties"]["forecast"]
        forecast = http.get(forecast_url, headers=headers, timeout=30.0).json()

    periods = forecast["properties"]["periods"][:3]
    return "\n".join(
        f"{p['name']}: {p['temperature']}°{p['temperatureUnit']}, "
        f"{p['detailedForecast']}"
        for p in periods
    )

def run_agent(user_message: str):
    messages = [{"role": "user", "content": user_message}]
    response = client.chat.completions.create(
        model="gpt-4o", messages=messages, tools=tools,
    )
    tool_call = response.choices[0].message.tool_calls[0]
    args = json.loads(tool_call.function.arguments)

    # Manual dispatch — we must map every tool name to a handler
    if tool_call.function.name == "get_forecast":
        result = get_forecast(**args)

    messages.append(response.choices[0].message)
    messages.append({
        "role": "tool",
        "tool_call_id": tool_call.id,
        "content": result,
    })
    return client.chat.completions.create(
        model="gpt-4o", messages=messages,
    ).choices[0].message.content

print(run_agent("What's the weather in San Francisco?"))

Notice the pain points:

  • The tool schema is hardcoded in the agent — add a new tool and you rewrite the agent.
  • The API call logic (get_forecast) is tightly coupled to the NWS API format, headers, and URL structure.
  • The dispatch logic (if tool_call.function.name == ...) is manual and grows linearly with each tool.
  • Want to add GitHub search? That's a whole new function, a new schema definition, and another branch in the dispatch.

This works. But it doesn't scale. Now let's build something better.

Building the MCP Server

Step 1 — Project Setup

Initialize the project

We'll use uv (a fast Python package manager) and the official mcp SDK. The SDK ships with FastMCP, a high-level framework that lets you define tools with simple decorators.

terminal Shell 
# Create project and install dependencies
mkdir weather-mcp-server && cd weather-mcp-server
uv init && uv add "mcp[cli]" httpx

This gives us the MCP SDK (which includes FastMCP) and httpx for async HTTP requests. Python 3.10+ is required.

Step 2 — Implement the Server

The full server

Here's the complete, working MCP server. It wraps the same NWS API we used above, but now it's a standalone, reusable server that any MCP-compatible client can discover and use:

server.py Python 
from typing import Any
import httpx
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather")

NWS_API_BASE = "https://api.weather.gov"
USER_AGENT = "weather-mcp-server/1.0"


async def make_nws_request(url: str) -> dict[str, Any] | None:
    """Make a request to the NWS API with error handling."""
    headers = {"User-Agent": USER_AGENT, "Accept": "application/geo+json"}
    async with httpx.AsyncClient() as client:
        try:
            response = await client.get(url, headers=headers, timeout=30.0)
            response.raise_for_status()
            return response.json()
        except (httpx.HTTPError, ValueError):
            return None


@mcp.tool()
async def get_forecast(latitude: float, longitude: float) -> str:
    """Get weather forecast for a location.

    Args:
        latitude: Latitude of the location
        longitude: Longitude of the location
    """
    points_url = f"{NWS_API_BASE}/points/{latitude},{longitude}"
    points_data = await make_nws_request(points_url)

    if not points_data:
        return "Unable to fetch forecast data for this location."

    forecast_url = points_data["properties"]["forecast"]
    forecast_data = await make_nws_request(forecast_url)

    if not forecast_data:
        return "Unable to fetch detailed forecast."

    periods = forecast_data["properties"]["periods"]
    return "\n---\n".join(
        f"{p['name']}:\n"
        f"  Temperature: {p['temperature']}°{p['temperatureUnit']}\n"
        f"  Wind: {p['windSpeed']} {p['windDirection']}\n"
        f"  {p['detailedForecast']}"
        for p in periods[:5]
    )


@mcp.tool()
async def get_alerts(state: str) -> str:
    """Get active weather alerts for a US state.

    Args:
        state: Two-letter US state code (e.g. CA, NY)
    """
    url = f"{NWS_API_BASE}/alerts/active/area/{state}"
    data = await make_nws_request(url)

    if not data or "features" not in data:
        return "Unable to fetch alerts or no alerts found."

    if not data["features"]:
        return "No active alerts for this state."

    alerts = []
    for feature in data["features"]:
        props = feature["properties"]
        alerts.append(
            f"Event: {props.get('event', 'Unknown')}\n"
            f"Area: {props.get('areaDesc', 'Unknown')}\n"
            f"Severity: {props.get('severity', 'Unknown')}\n"
            f"Description: {props.get('description', 'N/A')}"
        )
    return "\n---\n".join(alerts)


if __name__ == "__main__":
    mcp.run(transport="stdio")

Let's break down what's happening:

  • FastMCP("weather") creates a named server instance.
  • The @mcp.tool() decorator registers a function as an MCP tool. The SDK automatically generates the JSON Schema from the function signature and docstring — no manual schema definition needed.
  • make_nws_request is a plain helper function, not an MCP concept. The tool functions are the public interface; implementation details stay private.
  • mcp.run(transport="stdio") starts the server using standard I/O, which is how most MCP hosts (Claude Desktop, Cursor) communicate with local servers.

Key difference from the old way: The server knows nothing about which AI model will use it. It doesn't import OpenAI, doesn't define function-calling schemas, doesn't handle dispatch. It just exposes tools and lets the protocol handle everything else.

Step 3 — Register with an AI Host

Connect it to Claude Desktop

To use our server in Claude Desktop, we add a single entry to the configuration file. No code changes to Claude, no SDK integration — just a config line:

claude_desktop_config.json JSON 
{
  "mcpServers": {
    "weather": {
      "command": "uv",
      "args": ["--directory", "/path/to/weather-mcp-server", "run", "server.py"]
    }
  }
}

That's it. Claude will launch the server, discover the get_forecast and get_alerts tools automatically, and use them when relevant. Ask "What's the weather in New York?" and Claude will call the right tool with the right arguments, without any custom wiring.

The Client Side — How AI Uses It

What if you're building your own AI agent and want to connect to MCP servers programmatically? The SDK provides a client interface that makes this straightforward:

client.py Python 
import asyncio
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def main():
    server_params = StdioServerParameters(
        command="uv",
        args=["--directory", "/path/to/weather-mcp-server", "run", "server.py"],
    )

    async with stdio_client(server_params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # Discover available tools — no hardcoded knowledge needed
            tools = await session.list_tools()
            print("Available tools:")
            for tool in tools.tools:
                print(f"  - {tool.name}: {tool.description}")

            # Call a tool by name
            result = await session.call_tool(
                "get_forecast",
                arguments={"latitude": 37.7749, "longitude": -122.4194},
            )
            print(f"\nForecast:\n{result.content[0].text}")

asyncio.run(main())

The critical line is session.list_tools(). The client doesn't need to know what tools exist ahead of time — it discovers them at runtime. This is the same mechanism AI hosts use: they query each connected MCP server for its capabilities and present them to the model as available functions.

And here's the payoff: this same client code works with any MCP server. Swap the weather server for a GitHub server, a database server, or a file-system server — the connection, discovery, and calling logic is identical. The protocol abstracts away the differences.

Side-by-Side Comparison

Traditional API MCP
What it is A contract for software-to-software communication A protocol for AI-to-tool communication
Tool discovery Read docs, hardcode schemas in your app AI auto-discovers tools and schemas at runtime
Adding a new tool Write new function + schema + dispatch logic in the agent Deploy a new MCP server, add one config line
Auth & transport Different per API (OAuth, API keys, headers...) Standardized (stdio, SSE, or streamable HTTP)
Coupling Agent ↔ API are tightly coupled Agent ↔ Server are fully decoupled
Reusability Integration code lives inside one app Any MCP-compatible host can use the same server
Scaling to N tools O(N) integration code in the agent O(1) — the protocol handles it
Uses APIs? — Yes — MCP servers typically wrap existing APIs

Important nuance: MCP does not replace APIs. APIs are the underlying mechanism that services expose. MCP is a layer on top that standardizes how AI models discover and interact with those APIs. An MCP server is essentially a universal adapter between the AI protocol and any backend service.

Key Takeaways

Building this weather MCP server took about 70 lines of Python. Once deployed, it can be used by Claude Desktop, Cursor, any custom agent, or any future MCP-compatible host — without changing a single line.

Here's what I think matters most about MCP as an AI engineer:

  1. It solves the M×N problem. Instead of every app integrating with every tool, MCP gives us a shared interface. M apps + N servers = M+N integrations, not M×N.
  2. It shifts work from the agent to the server. The complexity of interacting with an external service lives in the MCP server, not in the AI application. This makes agents simpler and more robust.
  3. It enables a tools ecosystem. Just as package managers let developers share libraries, MCP enables sharing of AI capabilities. Build a server once, and the entire ecosystem can use it.
  4. It's the "USB-C moment" for AI. Before USB-C, you needed a different cable for every device. MCP is bringing that same kind of convergence to AI tool integration — one standard to connect everything.

The bottom line

APIs let software talk to software.
MCP lets AI talk to everything — through a single, universal protocol.

It doesn't replace APIs. It gives AI a standard way to use them.
